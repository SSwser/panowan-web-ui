# Backend Runtime Bundle Guide

This guide explains how backend-local runtime bundles are stored, rebuilt, and verified under the ADR 0006 contract.

Use this document when deciding:

- which backend-local files belong in this repository,
- which files should stay ignored and disposable,
- how install, verify, and rebuild differ,
- and which acquisition strategy a backend should follow.

## 1. Quick Rule

Keep backend-local source of truth in Git. Ignore generated runtime output and ephemeral setup artifacts.

Backend-local source of truth includes:

- `backend.toml`,
- backend-root integration files such as `runner.py`,
- repo-owned runtime inputs under `sources/`,
- and durable backend-root metadata such as `requirements.txt`.

Generated runtime output and ephemeral setup artifacts include:

- `vendor/`,
- `vendor/.revision`,
- `.tmp/`,
- `build/`,
- and temporary acquisition trees used only during setup.

## 2. Four Backend Layers

ADR 0006 defines four semantic layers inside a backend root.

### 2.1 Project integration layer

Examples:

- `runner.py`
- backend-root helper modules that define project-owned invocation logic

These files explain how this repository invokes the backend runtime. They are committed to Git and must survive rebuild.

### 2.2 Repo-owned runtime input layer

Example:

- `sources/**`

These files are committed to Git and participate in materialization. They are not disposable output.

Important: `sources/` is not a mirror of the full upstream repository. It is the place for project-owned runtime inputs that must be merged into the generated bundle.

### 2.3 Generated runtime bundle layer

Example:

- `vendor/**`
- `vendor/.revision`

These files are generated by setup tooling and consumed by runtime code. They remain ignored and are always safe to delete and rebuild.

### 2.4 Ephemeral setup layer

Examples:

- `.tmp/**`
- `build/**`
- temporary clone or extraction directories

These files exist only to support setup work and remain ignored.

## 3. What Stays in Git

Keep these backend-local files in Git:

- `backend.toml`
- `runner.py`
- `sources/**`
- backend-root metadata such as `requirements.txt`
- any additional durable backend-local integration or metadata files

### Why

These files are the declared inputs needed to regenerate a runtime bundle faithfully.

If deleting a file would lose project-owned runtime knowledge that install cannot recover from upstream source alone, that file belongs in Git.

## 4. What Stays Ignored and Disposable

Ignore these backend-local files:

- `vendor/**`
- `vendor/.revision`
- `.tmp/**`
- `build/**`
- temporary acquisition trees

### Why

These files are derived state. They exist so runtime can execute or setup can proceed, but they are not the durable source of truth.

A disposable file or directory must be reproducible from declared inputs.

## 5. Install, Verify, Rebuild

### 5.1 Install

Install ensures that backend runtime contract is satisfied.

Typical flow:

1. acquire upstream source for declared revision,
2. narrow it to declared runtime-relevant paths,
3. merge repo-owned runtime inputs from `sources/`,
4. materialize final runtime bundle into `vendor/`,
5. write provenance marker such as `.revision`,
6. verify expected files exist.

### 5.2 Verify

Verify is read-only.

It checks whether runtime bundle already satisfies the declared contract:

- `vendor/` exists,
- provenance marker matches,
- expected runtime files exist.

Verify must not mutate state.

### 5.3 Rebuild

Rebuild forces regeneration of runtime bundle from declared inputs.

Rebuild may delete and recreate `vendor/` because generated runtime output is explicitly disposable.

## 6. Acquisition Strategy Taxonomy

Different backends can acquire upstream source differently while still converging on the same backend-local runtime contract.

### 6.1 Strategy A: owned full source

Use this when project intentionally owns or co-maintains the full source tree.

Example:

- `third_party/PanoWan/` managed as a git submodule

Characteristics:

- full checkout is acceptable,
- source tree is intentionally preserved,
- long-lived ownership is explicit.

### 6.2 Strategy B: selective transient acquisition

Use this when upstream repository is large and only a small runtime subset is needed.

Examples:

- SeedVR-like repositories with training code, datasets, notebooks, or large non-runtime trees

Characteristics:

- acquire upstream source during install or rebuild,
- prefer shallow clone, sparse checkout, or equivalent narrowing when useful,
- apply include/exclude filtering to select runtime subset,
- merge repo-owned `sources/`,
- materialize only final runtime bundle into `vendor/`,
- delete temporary acquisition tree afterward.

Large upstream repositories should not be committed into this repository merely to support backend materialization.

### 6.3 Strategy C: archive or snapshot acquisition

Use this later if a backend is better served by a pinned archive than by normal git acquisition.

Characteristics:

- acquisition uses a revision-pinned archive or snapshot,
- extracted contents are filtered and materialized the same way,
- backend-local `sources/` and `vendor/` semantics stay unchanged.

This is a future option, not a current requirement.

## 7. Example Mapping

### PanoWan

- role: project-maintained engine source
- likely strategy: owned full source
- implication: full source tree can remain preserved intentionally

### RealESRGAN

- role: backend-local runtime bundle under `third_party/Upscale/realesrgan/`
- current migration target: replace historical `overlay` terms with `sources/`
- implication: keep repo-owned runtime inputs in Git, keep `vendor/` ignored

### SeedVR-like backend

- role: future large upstream backend
- likely strategy: selective transient acquisition
- implication: do not commit full upstream repository into this repository solely for setup purposes

## 8. Maintainer Checklist

When reviewing a backend layout, ask:

1. Does `runner.py` or equivalent backend-root integration code live outside generated output?
2. Are repo-owned runtime inputs under `sources/` and committed to Git?
3. Is `vendor/` treated as generated output and ignored?
4. Are temporary setup directories ignored and safe to delete?
5. Does `backend.toml` describe provenance, filtering, output target, and expected runtime bundle shape?
6. Can install or rebuild regenerate disposable output from declared inputs without relying on manual edits inside `vendor/`?

If any answer is no, the backend likely drifts from ADR 0006.
