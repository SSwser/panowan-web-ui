# I2V (Image-to-Video) Implementation Plan

> **For agentic workers:** Use superpowers:subagent-driven-development or superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add image-guided 360° video generation to panowan-worker. Users upload an image (or provide a URL); the worker generates a panoramic video seeded by that image's content.

**Architecture decision:** Use the existing Wan2.1-T2V-1.3B + PanoWan LoRA via the **video-to-video** (`input_video`) pathway of `WanVideoPipeline`. The image is encoded as a single-frame video and passed to the pipeline with a `denoising_strength < 1.0` so the output respects the input composition. This is the same zero-shot mechanism used by PanoWan's "Video Outpainting" demo. True CLIP-conditioned i2v (requiring Wan2.1-I2V-14B + new LoRA training) is tracked as a separate future task.

**Why not Wan2.1-I2V-14B?** See the spec for full analysis. Short answer: the PanoWan LoRA (trained on T2V-1.3B) is architecturally incompatible with I2V-14B (different parameter count), and retraining is a separate research effort.

**Tech Stack:** FastAPI multipart/form-data, Pillow, imageio[ffmpeg], existing subprocess pipeline, vanilla HTML/CSS/JS.

## Background: How the CLI receives `input_video`

`panowan-test` does **not** currently expose `--input-video`. The worker will invoke the pipeline Python API directly (bypassing the CLI) for i2v jobs, using a thin Python wrapper script:

```
/app/PanoWan/run_i2v.py  (generated at container build time or on first use)
```

This script imports `WanVideoPipeline` directly, passes `input_video`, and saves the output. It accepts the same flags as `panowan-test` plus `--input-video-path` and `--denoising-strength`. See the spec for the full script.

```python
"""Image-to-video helpers: image intake and single-frame video preparation."""
import base64
import os
import uuid
from io import BytesIO
from typing import Optional

import imageio
import numpy as np
from PIL import Image

from .settings import settings


def extract_image(payload: dict) -> Optional[bytes]:
    """Return raw image bytes from payload['image'] (base64 str) or None."""
    raw = payload.get("image")
    if not raw:
        return None
    if isinstance(raw, str):
        # Strip data-URI prefix if present
        if "," in raw:
            raw = raw.split(",", 1)[1]
        return base64.b64decode(raw)
    if isinstance(raw, bytes):
        return raw
    return None


def prepare_i2v_input(image_bytes: bytes, width: int, height: int) -> str:
    """
    Convert raw image bytes to a single-frame MP4 sized (width × height).
    Returns the path to the temporary video file.
    """
    os.makedirs(settings.i2v_upload_dir, exist_ok=True)
    img = Image.open(BytesIO(image_bytes)).convert("RGB").resize((width, height))
    frame = np.array(img)  # (H, W, 3) uint8
    out_path = os.path.join(settings.i2v_upload_dir, f"i2v_input_{uuid.uuid4()}.mp4")
    with imageio.get_writer(out_path, fps=15, quality=9) as writer:
        writer.append_data(frame)
    return out_path
```

## Future: True CLIP-conditioned I2V (Wan2.1-I2V-14B)

The `WanVideoPipeline` already has `encode_image()` and the `input_image` code path (guarded by `self.image_encoder is not None`). Enabling it requires:

1. Download Wan2.1-I2V-14B-480P (~30 GB)
2. Train a new PanoWan LoRA on the I2V-14B backbone (separate research)
3. Add `--input-image` to `panowan-test` CLI (upstream PR)
4. Update model load path to include the CLIP image encoder weights
5. Update worker to pass `input_image` instead of `input_video`

This is a separate roadmap item. The V2V approach above delivers useful image-guided generation with no new model weights.
