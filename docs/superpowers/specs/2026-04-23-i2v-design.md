# I2V (Image-to-Video) Feature Design

## Overview

Add image-guided 360° panoramic video generation to panowan-worker. A user provides a reference image alongside a text prompt; the system generates a panoramic video whose content, color palette, and composition are seeded by that image.

---

## 1. Upstream Research Findings

### 1.1 `WanVideoPipeline` supports two forms of image input

Inspecting `src/diffsynth/pipelines/wan_video.py` (commit `462ee29`):

```python
def __call__(
    self,
    prompt,
    negative_prompt="",
    input_image=None,   # ← True CLIP-conditioned i2v
    input_video=None,   # ← Video-to-video (img as first frame)
    denoising_strength=1.0,
    ...
):
```

**`input_image` path** — when `input_image is not None and self.image_encoder is not None`:
- Calls `encode_image()`: CLIP-embeds the image via `WanImageEncoder`, VAE-encodes it as first frame
- Injects `clip_feature` + `y` tensors into every DiT forward pass
- This is the full Wan2.1-I2V conditioning mechanism

**`input_video` path** — when `input_video is not None`:
- VAE-encodes the video, then adds noise at timestep[0] scaled by `denoising_strength`
- Equivalent to img2img but in latent video space
- Works with the existing T2V-1.3B model + PanoWan LoRA (no image encoder needed)

### 1.2 `panowan-test` CLI exposes neither path

`src/diffsynth/scripts/test.py` options:
```
--wan-model-path, --lora-checkpoint-path, --output-path,
--prompt, --negative-prompt, --num-inference-steps,
--seed, --tiled, --width, --height
```

No `--input-image` or `--input-video` flag exists.

### 1.3 True i2v requires Wan2.1-I2V-14B — incompatible with current LoRA

| Aspect | T2V-1.3B (current) | I2V-14B |
|--------|-------------------|---------|
| Image encoder | ❌ None | ✅ `WanImageEncoder` (CLIP ViT) |
| DiT size | 1.3B params | 14B params |
| PanoWan LoRA compatible | ✅ Yes | ❌ No (different arch) |
| Model weight size | ~6 GB | ~30 GB |
| Requires new LoRA training | No | Yes (research effort) |

`model_config.py` already registers `wan_video_image_encoder` as a known model type — the infrastructure exists, but the weights and a compatible LoRA do not.

### 1.4 Conclusion: Use `input_video` V2V as practical i2v

Converting the image to a single-frame video and running the pipeline with `denoising_strength ∈ [0.7, 0.95]` gives content-aware panoramic generation:
- No new model download
- No LoRA retraining
- Full ERP warp + circular padding + latent rolling still applies
- Same mechanism PanoWan uses for its "Video Outpainting" demo

## 2. Architecture

```
User uploads image (JPEG/PNG/WEBP)
      │
      ▼
POST /generate/i2v  (multipart/form-data)
      │
      ├─ Validate MIME type
      ├─ Save to i2v_upload_dir / {job_id}_input.{ext}
      ▼
prepare_i2v_input(image_bytes, width, height)
      │  PIL resize → numpy frame → imageio single-frame MP4
      ▼
  input_video_path  ──► stored in job payload
      │
      ▼
_run_generation_job (background)
      │
      ├─ generate_video(payload)
      │     ├─ detects input_video_path in payload
      │     ├─ _ensure_i2v_script(panowan_dir)   ← writes run_i2v.py if absent
      │     └─ subprocess: uv run python run_i2v.py  ...  --input-video-path ...
      ▼
  output_{job_id}.mp4
```

## 3. The `run_i2v.py` Wrapper Script

Because `panowan-test` CLI does not expose `--input-video`, the worker writes a minimal Python script into the PanoWan directory:

```python
# /app/PanoWan/run_i2v.py  (auto-generated by worker on first i2v job)
"""Thin wrapper: runs WanVideoPipeline with input_video for image-guided generation."""
import argparse
from pathlib import Path
import imageio
import numpy as np
import torch
from PIL import Image

from diffsynth.models.model_manager import ModelManager
from diffsynth.pipelines.wan_video import WanVideoPipeline
from diffsynth.data import save_video

parser = argparse.ArgumentParser()
parser.add_argument("--wan-model-path", required=True)
parser.add_argument("--lora-checkpoint-path", required=True)
parser.add_argument("--output-path", required=True)
parser.add_argument("--prompt", required=True)
parser.add_argument("--negative-prompt", default="")
parser.add_argument("--input-video-path", required=True)
parser.add_argument("--denoising-strength", type=float, default=0.85)
parser.add_argument("--num-inference-steps", type=int, default=50)
parser.add_argument("--width", type=int, default=896)
parser.add_argument("--height", type=int, default=448)
parser.add_argument("--seed", type=int, default=0)
parser.add_argument("--tiled", type=bool, default=True)
args = parser.parse_args()

wan_model_path = Path(args.wan_model_path)
model_manager = ModelManager(device="cpu")
model_manager.load_models([
    str(wan_model_path / "diffusion_pytorch_model.safetensors"),
    str(wan_model_path / "models_t5_umt5-xxl-enc-bf16.pth"),
    str(wan_model_path / "Wan2.1_VAE.pth"),
], torch_dtype=torch.bfloat16)
model_manager.load_lora(args.lora_checkpoint_path, lora_alpha=1.0)

pipe = WanVideoPipeline.from_model_manager(model_manager, torch_dtype=torch.bfloat16, device="cuda")
pipe.enable_vram_management(num_persistent_param_in_dit=None)

# Load single-frame video as input
reader = imageio.get_reader(args.input_video_path)
frames = [Image.fromarray(frame) for frame in reader]
reader.close()

video = pipe(
    prompt=args.prompt,
    negative_prompt=args.negative_prompt,
    input_video=frames,
    denoising_strength=args.denoising_strength,
    num_inference_steps=args.num_inference_steps,
    width=args.width,
    height=args.height,
    seed=args.seed,
    tiled=args.tiled,
)

output_path = Path(args.output_path)
output_path.parent.mkdir(parents=True, exist_ok=True)
save_video(video, str(output_path), fps=15, quality=10, ffmpeg_params=["-crf", "18"])
print(f"Video saved to {output_path}")
```

`_ensure_i2v_script()` in `generator.py` writes this file once and caches it:

```python
_I2V_SCRIPT_PATH: str | None = None

def _ensure_i2v_script(panowan_dir: str) -> str:
    global _I2V_SCRIPT_PATH
    script_path = os.path.join(panowan_dir, "run_i2v.py")
    if not os.path.exists(script_path):
        with open(script_path, "w") as f:
            f.write(I2V_SCRIPT_CONTENT)  # module-level constant string
    _I2V_SCRIPT_PATH = script_path
    return script_path
```

## 5. API

### New: `POST /generate/i2v`

**Request:** `multipart/form-data`

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `image` | file | ✅ | — | JPEG, PNG, or WEBP; max 20 MB |
| `prompt` | string | ✅ | — | Scene description |
| `negative_prompt` | string | | `""` | |
| `num_inference_steps` | int | | 50 | |
| `width` | int | | 896 | Must be 2× height |
| `height` | int | | 448 | |
| `seed` | int | | 0 | |
| `denoising_strength` | float | | 0.85 | 0.5 (more image) – 1.0 (pure T2V) |

**Validation:**
- `image` MIME type must be `image/jpeg`, `image/png`, or `image/webp`
- `image` size ≤ 20 MB
- `width == height * 2` (panoramic constraint, same as T2V)
- `denoising_strength` ∈ [0.0, 1.0]

**Response (202):**
```json
{
    "job_id": "7ddf135f-...",
    "status": "queued",
    "mode": "i2v",
    "prompt": "...",
    "download_url": "/jobs/7ddf135f-.../download"
}
```

**Errors:**
- `415 Unsupported Media Type` — unsupported image format
- `413 Content Too Large` — image > 20 MB
- `422 Unprocessable Entity` — missing required fields or constraint violation

### `GET /jobs` — updated response

Each job record now includes `mode` (`"t2v"` or `"i2v"`) and optionally `input_image_path`.

### Existing endpoints — unchanged

`POST /generate`, `GET /jobs/{id}`, `GET /jobs/{id}/download`, `GET /health`, SSE/cancel (if implemented) are all unaffected.

## 9. Security Considerations

- **File upload validation**: check `Content-Type` header AND inspect magic bytes (first 8 bytes) before passing to Pillow. Prevents MIME-type spoofing.
- **File size cap**: enforce 20 MB at the API layer before reading body.
- **Temp file cleanup**: uploaded images and single-frame input videos are retained until their parent job is garbage-collected (same lifecycle as output files). No automatic deletion to allow re-inspection.
- **Path traversal**: save upload with a UUID-keyed filename; never use client-provided filenames.
- **No shell injection**: subprocess command built from validated numeric parameters and fixed paths; no f-string interpolation of user strings into the command list (prompt is passed as a list element, not interpolated into a shell string).

## 10. Future: True CLIP-conditioned I2V

When the PanoWan team trains a LoRA on Wan2.1-I2V-14B:

1. Add `--input-image` to upstream `panowan-test` CLI (or extend `run_i2v.py`)
2. Load image encoder weights from I2V-14B model directory
3. Change `prepare_i2v_input` → `prepare_i2v_clip`: returns `PIL.Image` directly
4. Pass `input_image=pil_image` instead of `input_video=frames` to pipeline
5. The `encode_image()` CLIP path in `WanVideoPipeline` will activate automatically

The worker API surface (`POST /generate/i2v`) stays identical — only the backend changes.
